# Working with issues

Tyler is non-technical. When he describes a change, a bug, or an idea in a
Claude session, turn it into a well-formed GitHub issue (don't silently start
coding unless he explicitly says "just do it"). Structure every issue:

- **Title** — short, plain-English, imperative. e.g. "Make show dates bigger and
  bolder on the Shows tab".
- **Body**, with these headings:
  - **What** — the change, in plain English.
  - **Where** — the screen/area affected (Home / Listen / Shows / Journal / Pod /
    onboarding / player bar).
  - **Acceptance criteria** — a short checklist describing what "done" looks like.
- **Labels** — apply at least one: `enhancement` (new/changed feature), `bug`
  (something broken), `documentation`, or `question`. Add `ios` / `android` /
  `web` if a change is platform-specific.

After creating an issue, reply with its number and link. Keep all issue language
non-technical.
